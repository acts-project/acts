The Kalman filter in ACTS is not a standalone loop — it is a plugin (an "Actor") that rides inside the track propagator. The propagator transports a track surface-by-surface; at each surface it calls back into the Kalman actor, which performs the measurement update. Keep that in mind and everything else follows.

1. What ACTS calls "the Kalman filter"

The Kalman filter appears in three related-but-distinct places:

This document focuses on the KalmanFitter and the shared math kernels (GainMatrixUpdater, GainMatrixSmoother, MbfSmoother) that the CKF and GSF reuse.

2. The big picture: the KF as a propagator Actor

The Propagator walks a track through the geometry using a stepper (B-field ODE integration, e.g. SympyStepper) and a navigator (next surface). It supports a list of Actors — callbacks invoked at each step. KalmanFitter defines a private Actor (KalmanFitter.hpp:286) injected into that list.

Propagator.propagate()
   └─ for each step:
        stepper integrates ODE to next surface
        navigator identifies currentSurface
        Actor::act(state, stepper, navigator, result)   ← the Kalman logic
        Actor::checkAbort(...)                           ← stop when done

A "fit" is therefore a propagation with a measurement-updating actor attached. The fitter object (KalmanFitter.hpp:252) is thin: it owns a propagator + loggers, sets up options, runs the propagation, then orchestrates smoothing. It is templated on <propagator_t, traj_t>; a static constexpr bool isDirectNavigator (:258) selects the standard Navigator vs. the DirectNavigator (explicit surface sequence, used for refits).

3. The data structures that flow through a fit

  • KalmanFitterOptions (:108-186) — the three contexts (geometry/field/calibration), the extensions, propagator options, target referenceSurface + strategy, physics toggles (multipleScattering, energyLoss, freeToBoundCorrection), and smoothing controls (reverseFiltering, reverseFilteringCovarianceScaling default 100).
  • KalmanFitterExtensions (:43-103) — six Acts::Delegates: calibrator, updater, smoother, outlierFinder, reverseFilteringLogic, surfaceAccessor. Defaults are void stubs.
  • KalmanFitterResult (:189-229) — the scratchpad: fittedStates, tip indices, counters, missedActiveSurfaces, fittedParameters, finished.

4. The fit() entry point and the two-pass structure

Public fit(...) overloads (:667, :695) forward to fit_impl (:835):

fit_impl
 ├─ make_propagator_options(... reverseDirection=false)   build forward options
 ├─ filter_impl(...)                                       FORWARD FILTER PASS
 │     └─ propagator.propagate()  →  Actor runs at every surface
 ├─ seed last measurement: smoothed ← filtered
 ├─ decide: reverse-filter or smooth?
 │     ├─ reverse filtering:  filter_impl(... reverseDirection=true)
 │     └─ direct smoothing:   extensions.smoother(...)
 ├─ extrapolate to referenceSurface (if needed)
 └─ calculateTrackQuantities(track)

Two setup details in make_propagator_options (:710): (1) measurements are copied into an unordered_map<const Surface*, SourceLink> (:726-731) — so measurements need not be pre-sorted, ordering emerges from navigation; (2) each measurement surface is registered with the navigator as an external surface (:744-755). filter_impl fails the fit if zero measurement states result (:801-804, NoMeasurementFound).

5. The forward filter pass — Actor::act → Actor::filter

act (:339) runs every step: arms loop protection, calls filter if the navigator reports a current surface, then checks termination (track complete, end-of-world, volume/path limit, target reached → bind final parameters, set finished). filter (:445) branches on what the surface is: (A) carries a measurement, (B) a hole/material surface, or (C) nothing to record.

6. The Kalman update math — GainMatrixUpdater

The default updater delegate (GainMatrixUpdater.hpp, math in detail/GainMatrixUpdaterImpl.hpp). The measurement dimension N (1–6) is only known at runtime, so visit_measurement dispatches to a compile-time visitMeasurementImpl<N> (explicitly instantiated for N = 1..6).

With $x^-,P^-$ = predicted state & covariance, $m,R$ = calibrated measurement & covariance, and $H$ the projector from the 6-dim bound space to the N-dim measurement space:

$$ K = P^- H^\top \left( H P^- H^\top + R \right)^{-1} \qquad\text{(gain, :52)} $$

$$ x^+ = x^- + K\left( m - H x^- \right) \qquad\text{(filtered params, :65)} $$

$$ P^+ = (I - KH)\,P^- \quad\text{or, Joseph form:}\quad (I-KH)P^-(I-KH)^\top + K R K^\top \quad\text{(:69-75)} $$

$$ r = m - H x^+,\qquad \chi^2 = r^\top \big[(I - HK)R\big]^{-1} r \qquad\text{(:80-88)} $$

The gain $K$ is the crux: it is the weight that blends the prediction with the measurement. The widget below shows the idea in 1D (so $H=1$, all quantities scalar): $K = \dfrac{\sigma_p^2}{\sigma_p^2 + \sigma_m^2}$. Drag the sliders.

Notice: the filtered curve is always narrower than either input — adding a measurement always reduces uncertainty. When the measurement is very precise ($\sigma_m \to 0$), $K \to 1$ and the filtered mean snaps onto it; when it is very imprecise ($\sigma_m$ large), $K \to 0$ and the prediction is barely moved.

7. Smoothing — three ways to get the "best" estimate

The forward filter's filtered uses only past+present hits. Smoothing folds in future hits. The KF seeds the last measurement's smoothed ← filtered (:865-867) and propagates that backward. There are three strategies:

Interactive · compare smoothers

Rauch–Tung–Striebel. GainMatrixSmoother.cpp. Walks backward via applyBackwards; ts = state being smoothed, prev_ts = already-smoothed neighbour toward the end. prev_ts.jacobian() is the forward transport $F_k$ from $k$ to $k{+}1$.

$$ G = C^f_k\, F_k^\top \left(P^-_{k+1}\right)^{-1} $$

$$ x^s_k = x^f_k + G\left(x^s_{k+1} - x^p_{k+1}\right) $$

$$ C^s_k = C^f_k + G\left(C^s_{k+1} - P^-_{k+1}\right) G^\top $$

Cost: inverts the full $6\times6$ predicted covariance at every step. Optional positive-definiteness check.

Modified Bryson–Frazier — the default in the examples (KalmanFitterFunction.cpp:77). Propagates two adjoint accumulators $\hat\Lambda$ (6×6) and $\hat\lambda$ (6-vec), both starting at zero — no full-covariance inverse needed, only the small N×N residual covariance.

$$ x^s = x^f - C^f\,\hat\lambda, \qquad C^s = C^f - C^f\,\hat\Lambda\,C^f $$

On a measurement state, with $S=HP^-H^\top+R$, $K=P^-H^\top S^{-1}$, $\hat C = I-KH$:

$$ \tilde\Lambda = H^\top S^{-1} H + \hat C^\top \hat\Lambda \hat C,\qquad \tilde\lambda = -H^\top S^{-1} y + \hat C^\top \hat\lambda $$

then transport: $\hat\Lambda = F^\top \tilde\Lambda F$, $\hat\lambda = F^\top\tilde\lambda$. Non-measurement states only transport.

Cheaper and numerically nicer than RTS — hence the default.

Reverse filtering. Chosen when reverseFiltering is set or reverseFilteringLogic(...) returns true (:871-874) — typically for low-momentum tracks where linearised smoothing is inaccurate. Instead of an analytic backward sweep, it runs the whole filter again backward (:882-885), seeded from the forward last measurement with covariance inflated ×reverseFilteringCovarianceScaling (default 100, :880-881) so forward information does not dominate. Costs a second full propagation.

8. Finishing up — reference surface & track quantities

If parameters are not already on the requested referenceSurface, a final extrapolateTrackToReferenceSurface is done (:933-946). Then calculateTrackQuantities (TrackHelpers.hpp:388) sweeps states in reverse: counts holes/outliers/measurements/shared hits and sums only measurement states' $\chi^2$ into track.chi2() and their sizes into track.nDoF() (:399-414).

9. The extension points (and why they're delegates)

Flexibility comes from the six Acts::Delegates (type-erased, allocation-free, connected at runtime). Default void stubs live in detail/VoidFitterComponents.hpp: voidFitterUpdater copies predicted→filtered; voidFitterSmoother copies filtered→smoothed; voidOutlierFinder never flags; voidReverseFilteringLogic never reverses; voidFitterCalibrator / voidSurfaceAccessor throw (you must supply real ones). This is what lets the same KalmanFitter code drive a pixel detector, a straw-tube tracker, an alignment refit, or a unit test — only the connected functions change.

10. How it's actually wired up (the Examples layer)

Examples/Algorithms/TrackFitting/src/KalmanFitterFunction.cpp shows production wiring: stepper = SympyStepper; two fitters (standard + direct navigator). makeKfOptions connects GainMatrixUpdater as updater, MbfSmoother as smoother, a momentum-gated SimpleReverseFilteringLogic, and a χ²-cut SimpleOutlierFinder; useJosephFormulation flows into the updater. So a user's "Kalman fitter" is: SympyStepper + Navigator + KalmanFitter actor + GainMatrixUpdater + MbfSmoother.

Six things to remember.
  1. The KF is a propagator actor; navigation drives surface ordering, so measurements need not be sorted.
  2. Every surface yields a track state with predicted/filtered/smoothed slots and type flags (measurement/outlier/hole/material).
  3. The gain-matrix update is the recursive forward step; the stepper is re-seeded with the filtered state.
  4. Material is modelled inline via pre/post-update interactions.
  5. Smoothing runs future info backward — cheaply (MBF/RTS) or robustly (reverse filter).
  6. Everything pluggable is a Delegate — the same code serves fitting, refitting, the CKF, and the GSF.
